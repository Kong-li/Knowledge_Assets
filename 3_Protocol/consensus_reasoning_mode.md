# 《共识推理模式协议》（Consensus Reasoning Mode Protocol）

---

## 1. 协议目的

本协议定义**共识推理模式**的完整行为规范，用于解决以下核心问题：

> 当您说"用系统思维分析"时，如何让AI使用**您的定义**而非**互联网平均定义**？

### 1.1 要解决的问题

**问题场景：**
```
用户："用系统思维来分析这个问题"

AI的困境：
- 网络上"系统思维"定义混乱
- 不同书籍、不同学者有不同解释
- AI默认会综合多种来源，生成"平均意义"的理解

但用户真正想要的是：
- 用户读过的那本书的定义
- 用户已经加工、确认、内化的理解
- 用户在认知资产中保存的专属定义
```

**本质问题：** 概念共识的缺失导致交流基础不稳定。

---

## 2. 模式定义

### 2.1 什么是共识推理模式

**定义：**
> 共识推理模式是一种认知协作姿态，在此模式下，AI必须基于用户已确认的认知资产进行推理，将资产中的定义、框架、方法论视为**唯一权威来源**。

### 2.2 核心特征

```
认知基础：用户的认知资产
推理约束：资产定义 > 网络通识
语义来源：显式引用的资产
灵活度：  在共识约束下的自由推理
```

### 2.3 与其他模式的区别（未来）

| 维度 | 共识推理模式 | 自由探索模式（未定义） | 严格执行模式（未定义） |
|------|-------------|---------------------|---------------------|
| 资产依赖 | ✅ 必需 | ❌ 不使用 | ✅ 强制 |
| 推理自由度 | 中 | 高 | 低 |
| 概念来源 | 用户资产 | 网络通识 | 用户SOP |

## 3. 触发与激活

### 3.1 触发关键词

用户在提示词中使用：

```
【共识推理模式】
```

**示例：**
```
【共识推理模式】使用[[系统思维]]分析团队效率低下的问题
```

### 3.2 激活条件

模式成功激活需要满足：

1. ✅ 用户显式使用触发关键词
2. ✅ 提示词中包含至少一个资产引用 `[[资产名]]`
3. ✅ 引用的资产在注册表中存在

### 3.3 激活失败处理

#### 情况1：未包含资产引用

```
用户："【共识推理模式】分析这个问题"

AI响应：
"您已激活共识推理模式，但未指定使用哪些认知资产。
请使用 [[资产名]] 语法引用相关资产，例如：
【共识推理模式】使用[[系统思维]]分析这个问题"
```

#### 情况2：引用的资产不存在

```
用户："【共识推理模式】使用[[系统思维]]分析问题"

AI检查后：
1. 访问Knowledge注册表
2. 发现不存在名为"系统思维"的资产

AI响应：
"资产 [[系统思维]] 在您的认知资产库中不存在。
当前可用的相关资产有：
- [[个人期望和时代需求]]
您可以：
1. 调整引用的资产名称
2. 创建该资产后再使用
3. 切换到自由探索模式"
```

## 4. AI的执行流程

### 4.1 标准执行流程

```
┌─────────────────────────────────────┐
│ Step 1: 识别触发关键词               │
│ 【共识推理模式】                     │
└──────────────┬──────────────────────┘
               ↓
┌─────────────────────────────────────┐
│ Step 2: 解析资产引用                 │
│ 提取所有 [[资产名]] 标记             │
└──────────────┬──────────────────────┘
               ↓
┌─────────────────────────────────────┐
│ Step 3: 加载结构规范协议             │
│ - 《认知资产标准格式协议》(CASFP)    │
│ - 《认知资产引用与链接协议》(CALP)   │
└──────────────┬──────────────────────┘
               ↓
┌─────────────────────────────────────┐
│ Step 4: 访问注册表                   │
│ 根据资产类型访问对应注册表           │
└──────────────┬──────────────────────┘
               ↓
┌─────────────────────────────────────┐
│ Step 5: 定位并加载资产               │
│ 通过注册表中的URL加载资产本体        │
└──────────────┬──────────────────────┘
               ↓
┌─────────────────────────────────────┐
│ Step 6: 解析资产内容                 │
│ - 提取元数据（YAML Front Matter）   │
│ - 解析正文结构                       │
│ - 识别资产内的二次引用               │
└──────────────┬──────────────────────┘
               ↓
┌─────────────────────────────────────┐
│ Step 7: 建立推理上下文               │
│ 将资产内容作为推理的语义背景         │
└──────────────┬──────────────────────┘
               ↓
┌─────────────────────────────────────┐
│ Step 8: 执行推理                     │
│ 基于用户资产定义进行分析             │
└─────────────────────────────────────┘
```

### 4.2 资产依赖处理

如果被引用的资产A中又引用了资产B：

```
用户引用：[[资产A]]
资产A内容中包含：[[资产B]]

AI行为：
1. 识别资产A对资产B的依赖
2. 根据引用类型决定加载策略：
   - 整体引用 `[[资产B]]` → 完整加载资产B
   - 概念引用（脚注式）→ 记录依赖但不必完整加载
3. 按照依赖图的拓扑顺序加载资产
```

**加载顺序原则：** 先加载被依赖的资产，再加载依赖它的资产。

## 5. 推理约束规则

### 5.1 核心约束

在共识推理模式下，AI必须遵守：

**约束1：定义权威性**
```
资产中的定义 > 网络通识
```

**示例：**
```
如果用户的[[系统思维]]资产定义：
"系统思维是识别要素、关系和边界的能力"

即使网络上大部分定义强调"整体性""动态性"，
AI也必须以用户的定义为准。
```

**约束2：概念一致性**
```
同一对话中，对同一概念的理解必须保持一致
```

如果用户引用了[[系统思维]]，那么在整个对话中：
- 所有提到"系统思维"的地方
- 都必须与资产中的定义保持一致
- 不能中途切换到其他定义

**约束3：语义溯源**
```
当使用资产中的概念时，应明确标注来源
```

**示例：**
```
错误表述：
"从系统思维角度来看..."

正确表述：
"根据您定义的[[系统思维]]（识别要素、关系和边界），..."
```

### 5.2 允许的自由度

虽然受资产约束，但AI仍可以：

- ✅ 在资产定义的框架内进行推理演绎
- ✅ 将资产中的方法应用到新场景
- ✅ 组合多个资产进行综合分析
- ✅ 指出资产定义的潜在局限性（但不擅自修改）

### 5.3 禁止的行为

- ❌ 用网络定义替代资产定义
- ❌ 混合使用资产定义和网络定义
- ❌ 在未引用的情况下假设用户的理解
- ❌ 擅自修改或"优化"资产中的定义

## 6. 处理定义冲突

### 6.1 资产定义 vs 网络通识冲突

**AI行为：**
```
1. 明确指出差异存在
2. 说明资产定义的内容
3. 简要说明网络通识的内容
4. 声明：将以资产定义为准
5. 继续基于资产定义推理
```

**示例对话：**
```
AI: "我注意到您的[[系统思维]]定义侧重'要素-关系-边界'，
     而网络上常见定义更强调'整体性'和'涌现'。
     在本次分析中，我将严格基于您的定义来展开。"
```

### 6.2 多个资产定义冲突

如果用户引用的两个资产对同一概念有不同定义：

```
[[资产A]] 中："创新是突破性改进"
[[资产B]] 中："创新是持续性优化"

AI行为：
1. 明确指出冲突
2. 暂停推理
3. 请求用户澄清：应该使用哪个定义
```

## 7. 输出规范

### 7.1 推理过程的透明化

在共识推理模式下，AI应该：

**明确标注资产来源：**
```
"基于您在[[系统思维]]中定义的三要素..."
"按照[[问题分析框架]]的第二步骤..."
```

**区分资产内容与推理演绎：**
```
资产内容：
- 系统思维包含要素、关系、边界（来自资产）

推理演绎：
- 因此在团队效率问题中，我们可以识别...（AI推理）
```

### 7.2 引用标记规范

当直接引用资产内容时，使用明确标记：

```markdown
根据您的[[系统思维]]定义：
> 系统思维是识别要素、关系和边界的能力

基于这个定义，我们可以分析...
```

## 8. 边界与限制

### 8.1 本模式不负责的内容

- ❌ 判断资产内容的"正确性"
- ❌ 评价资产定义是否"主流"
- ❌ 建议修改资产定义
- ❌ 在资产定义明显错误时自行纠正

### 8.2 本模式的适用边界

**适合：**
- 在已有认知框架下分析新问题
- 需要概念一致性的长期讨论
- 将理论知识应用到实践场景
- 跨对话保持认知连续性

**不适合：**
- 探索全新领域（应使用探索模式）
- 质疑已有定义本身（应使用批判模式）
- 严格执行既定流程（应使用执行模式）

## 9. 模式的退出

### 9.1 自动退出条件

当出现以下情况时，AI应提示退出本模式：

1. 用户连续提问不涉及已引用的资产
2. 话题转向资产未覆盖的领域
3. 用户明确表示想探索新的可能性

### 9.2 手动退出

用户可以随时声明：
```
【退出共识推理模式】
或
【切换到XXX模式】
```

## 11. 设计理念说明

### 11.1 为什么要"共识"

**问题的本质：**
```
传统AI对话：基于"通识"
└─ 优势：覆盖面广
└─ 劣势：缺少个性化、深度不足、定义飘移

共识推理：基于"专识"
└─ 优势：概念精确、深度可控、可累积
└─ 劣势：需要预先构建资产
```

**设计目标：** 让AI协作从"临时性对话"演进为"可累积的认知过程"。

### 11.2 为什么需要显式声明

如果不显式声明，AI面临：
- 不确定用户是否要使用自己的定义
- 不确定何时该调用资产库
- 容易误判用户意图

显式声明的好处：
- 用户有控制权
- 行为可预测
- 减少误判成本

### 11.3 为什么限制AI自由

**核心理念：**
```
共识推理模式 ≠ 限制AI能力
共识推理模式 = 让AI能力聚焦在用户的认知体系内
```

就像在特定规则下下棋：
- 规则不是限制，而是让游戏有意义的前提
- 资产定义不是束缚，而是深度对话的基础
